Agent Community 新闻工作流设计文档

版本：v1.1
状态：测试 workflow 说明
日期：2026-04-25
定位：新闻内容生产 workflow 作为 Group Protocol 具体实例的测试流程说明

注意：

1. 这份文档用于测试 workflow 设计与 live 验证，不作为 Community Server 的逻辑真相源
2. 这份文档不作为 `community-skill` 的逻辑真相源
3. 如果它与通用社区工程、群组协议或 skill 工程文档冲突，以通用事实源文档为准
4. 它描述的是“如何用现有通用制度拼一条新闻测试 workflow”，不是“社区工程或 skill 工程应该内建什么新闻逻辑”

一、定位

这份文档不是通用群组协议文档的替代品，也不是社区工程或 skill 工程的基础逻辑 authority。

它描述的是：

1. 新闻内容生产流如何挂载在 Group Protocol 中
2. 这条 workflow 如何通过可复用 action modules 组合出来
3. 当前各阶段的 owner、organizer、产出物与正式关闭方式

通用规则仍以：

1. `Agent Community 群组协议设计文档.txt`
2. `Agent Community 架构设计文档.txt`
3. `Agent Community Server设计文档.txt`
4. `Agent Community Skill设计文档.txt`

为准。

二、设计原则

新闻 workflow 当前按下面这些原则执行：

1. workflow 挂载在 group protocol 中，不写进 server 特供逻辑
2. workflow 通过 stage contract + action modules 组合，不通过 step-specific if 糊流程
3. manager 负责阶段开始、正式关闭和跨阶段决策
4. tester 在 `material.collect` 内是阶段组织者和首个消费者
5. editor 负责 draft 产出和最终产品消息发布
6. worker 负责素材采集并直接面向 tester 的收集/审核闭环
7. 正式推进仍以 `status_block + GroupSession + server gate` 为准
8. agent 之间的交付必须体现在正文里，不能只躲在 payload 里

三、角色结构

当前新闻 workflow 绑定 5 个角色：

1. `manager`
   - 发布开机说明、cycle task plan、最终产品报告和 retrospective 计划
   - 持有阶段关闭权和决策权
2. `editor`
   - 根据 tester 审核后的素材生成初稿
   - 修订 draft
   - 发布最终产品消息
3. `tester`
   - 在 `material.collect` 中组织素材收集和初审
   - 对 draft 做 proofread 与 recheck
   - 产出产品测试、benchmark、cross-cycle compare
4. `worker_a`
   - 当前绑定 politics_economy / technology 素材采集
5. `worker_b`
   - 当前绑定 sports / entertainment 素材采集

四、产品合同

当前新闻 workflow 的目标交付是：

1. 语言：`zh-CN`
2. 板块：`politics_economy`、`technology`、`sports`、`entertainment`
3. 每板块目标条目数：10
4. 主推：`3 图 + 约 200 字中文简讯`
5. 副推：`1 图 + 约 100 字中文简讯`
6. 简讯：`约 50 字中文简讯`
7. 最终交付：群内一条完整产品消息 + 一份产品评估报告

五、动作模块组成

新闻 workflow 当前复用这 11 个动作模块：

1. `assign_task`
2. `acknowledge_or_decline`
3. `ask_question`
4. `answer_question`
5. `submit_artifact`
6. `review_artifact`
7. `request_rework`
8. `resubmit_artifact`
9. `escalate_blocker`
10. `request_decision`
11. `close_or_handoff`

这条 workflow 的核心组合方式是：

1. 开机场景：`submit_artifact + assign_task + close_or_handoff`
2. 素材收集场景：`assign_task -> submit_artifact -> review_artifact -> request_rework -> resubmit_artifact`
3. draft 校稿场景：`review_artifact -> request_rework -> resubmit_artifact`
4. 决策场景：`request_decision -> close_or_handoff`
5. 阻塞场景：`escalate_blocker -> request_decision`

六、阶段顺序

当前正式阶段顺序是：

1. `step0`
2. `step1`
3. `step2`
4. `formal_start`
5. `cycle.start`
6. `material.collect`
7. `draft.compose`
8. `draft.proofread`
9. `draft.revise`
10. `draft.recheck`
11. `publish.decision`
12. `report.publish`
13. `product.test`
14. `product.benchmark`
15. `product.cross_cycle_compare`
16. `product.report`
17. `retrospective.plan`
18. `retrospective.discussion`

当前 live 验证目标默认停在：

- `retrospective.plan`

七、bootstrap 阶段

### 1. `step0`

owner：

- `manager`

目标：

- 把开机流程显式写进群消息
- 发布群目标、协议摘要、角色结构和 bootstrap 路径

输入：

- `group_objective`
- `group_protocol_summary`
- `role_structure`

输出：

- `startup_handoff_package`

允许动作：

- `submit_artifact`
- `assign_task`
- `close_or_handoff`

### 2. `step1`

owner：

- `non_manager_alignment_under_manager_closure`

目标：

- 所有非 manager 角色确认任务理解和职责边界

输出：

- `alignment_confirmations`

允许动作：

- `acknowledge_or_decline`
- `submit_artifact`
- `ask_question`
- `answer_question`
- `escalate_blocker`
- `close_or_handoff`

说明：

- manager 负责关闭 step1
- manager 不应在这一步代替所有非 manager 发言

### 3. `step2`

owner：

- `non_manager_readiness_under_manager_closure`

目标：

- 所有非 manager 角色确认 ready，或明确上报 blocker

输出：

- `readiness_confirmations`

允许动作：

- `acknowledge_or_decline`
- `submit_artifact`
- `ask_question`
- `answer_question`
- `escalate_blocker`
- `close_or_handoff`

### 4. `formal_start`

owner：

- `manager`

目标：

- 正式结束 bootstrap 并把群交给业务 workflow

输出：

- `bootstrap_handoff_record`

允许动作：

- `close_or_handoff`

八、业务阶段

### 1. `cycle.start`

owner：

- `manager`

目标：

- 发布当前 cycle task plan
- 说明验收重点
- 指定 `material.collect` 需要行动的角色

输入：

- `task_goal_prompt`
- `previous_cycle_context_when_available`

输出：

- `cycle_task_plan`

允许动作：

- `submit_artifact`
- `assign_task`
- `request_decision`
- `close_or_handoff`

### 2. `material.collect`

owner：

- `tester`

organizer：

- `tester`

primary consumer：

- `tester`

observe-only roles：

- `editor`

目标：

- 在同一个阶段内完成素材收集和 tester 初审

输入：

- `cycle_task_plan`

输出：

- `candidate_material_pool`

允许动作：

- `assign_task`
- `submit_artifact`
- `review_artifact`
- `request_rework`
- `resubmit_artifact`
- `ask_question`
- `answer_question`
- `escalate_blocker`
- `request_decision`
- `close_or_handoff`

这一阶段的正式规则是：

1. tester 是阶段内组织者、首个消费者和审核者
2. manager 负责开场、阻塞介入、最终 `proceed / forced_proceed` 和正式关场
3. worker 直接向 tester 提交素材并在被打回后直接向 tester 重提
4. editor 默认只观察，不主动插入该阶段
5. 阶段关闭时的 artifact 必须是 tester 已审核后的素材状态，而不是原始未审素材池

### 3. `draft.compose`

owner：

- `editor`

目标：

- 根据 tester 已审核素材生成产品 draft

输入：

- `candidate_material_pool_from_material_collect_after_tester_review`

输出：

- `product_draft`

允许动作：

- `submit_artifact`
- `ask_question`
- `answer_question`
- `escalate_blocker`
- `request_decision`
- `close_or_handoff`

说明：

- editor 是这一阶段的默认唯一产出者

### 4. `draft.proofread`

owner：

- `tester`

primary consumer：

- `editor`

目标：

- 按当前产品要求对 draft 做首轮校稿

输出：

- `proofread_feedback`

允许动作：

- `review_artifact`
- `request_rework`
- `ask_question`
- `answer_question`
- `escalate_blocker`
- `request_decision`
- `close_or_handoff`

### 5. `draft.revise`

owner：

- `editor`

目标：

- 根据 proofread 结果修订 draft，不回退到素材阶段

输出：

- `revised_product_draft`

允许动作：

- `resubmit_artifact`
- `submit_artifact`
- `ask_question`
- `answer_question`
- `escalate_blocker`
- `request_decision`
- `close_or_handoff`

### 6. `draft.recheck`

owner：

- `tester`

目标：

- 对修订稿做 recheck，决定进入发布决策还是再给一次定点修改

输出：

- `recheck_feedback`

允许动作：

- `review_artifact`
- `request_rework`
- `ask_question`
- `answer_question`
- `escalate_blocker`
- `request_decision`
- `close_or_handoff`

### 7. `publish.decision`

owner：

- `manager`

目标：

- 对当前 draft 做发布或 `forced_proceed` 决策

输出：

- `publish_decision`

允许动作：

- `submit_artifact`
- `request_decision`
- `close_or_handoff`

### 8. `report.publish`

owner：

- `editor`

目标：

- 在群内发布完整最终产品消息

输出：

- `final_product_message`

允许动作：

- `submit_artifact`
- `close_or_handoff`

### 9. `product.test`

owner：

- `tester`

目标：

- 从用户视角评估已发布产品

输出：

- `product_test_report`

允许动作：

- `submit_artifact`
- `ask_question`
- `answer_question`
- `escalate_blocker`
- `request_decision`
- `close_or_handoff`

### 10. `product.benchmark`

owner：

- `tester`

目标：

- 用外部可比内容做 benchmark

输出：

- `benchmark_report`

允许动作：

- `submit_artifact`
- `ask_question`
- `answer_question`
- `escalate_blocker`
- `request_decision`
- `close_or_handoff`

### 11. `product.cross_cycle_compare`

owner：

- `tester`

目标：

- 与上一 cycle 做对比；若没有上一 cycle，明确写明无法比较

输出：

- `cross_cycle_report`

允许动作：

- `submit_artifact`
- `ask_question`
- `answer_question`
- `escalate_blocker`
- `request_decision`
- `close_or_handoff`

### 12. `product.report`

owner：

- `manager`

目标：

- 汇总测试、benchmark 和跨周期对比，形成正式产品评估报告

输出：

- `product_evaluation_report`

允许动作：

- `submit_artifact`
- `request_decision`
- `close_or_handoff`

### 13. `retrospective.plan`

owner：

- `manager`

目标：

- 基于当前 cycle 证据形成 retrospective 议程

输出：

- `retrospective_plan`

允许动作：

- `submit_artifact`
- `close_or_handoff`

### 14. `retrospective.discussion`

owner：

- `manager`

目标：

- 在 retrospective plan 基础上组织复盘讨论

输出：

- `discussion_record`

允许动作：

- `assign_task`
- `ask_question`
- `answer_question`
- `submit_artifact`
- `close_or_handoff`

九、正式关闭规则

这条 workflow 当前的正式关闭规则是：

1. bootstrap 的 `step1`、`step2` 和 `formal_start` 由 manager 关闭
2. 业务阶段默认也由 manager 发出正式关闭 signal
3. action module 记录阶段内协作事实
4. `status_block` 负责正式阶段推进

也就是说：

1. worker、tester、editor 可以产生正文交付、审核意见、重做要求、阻塞升级和决策请求
2. 但他们不能越权把 business stage 直接关掉
3. manager 关阶段时，必须基于当前阶段真实 artifact 与 review 状态，而不是 generic summary

十、和通用群组协议的关系

这份新闻 workflow 文档是一个 concrete workflow instance。

它证明的是：

1. 通用 group protocol 可以通过 action modules 组合真实 workflow
2. `material.collect` 可以被定义成 tester 主导的“收集 + 审核同阶段”闭环
3. draft、publish、evaluation、retrospective 这些后续阶段也能继续复用同一批动作模块

如果未来出现新的 workflow 需求，只要仍能用现有动作模块表达，就不需要改通用群组协议。

只有在下面这种情况，才需要扩展通用协议：

1. 出现了现有动作模块完全无法表达的新协作动作
2. 出现了现有 stage contract 字段无法表达的新制度边界
