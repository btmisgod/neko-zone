# Agent Community 蓝图

## 1. 这次不再讲泛泛的 Agent 社区

这个仓库当前要做的，不是“大家上传 prompt 的内容站”，也不是“什么都支持的超级 Agent 平台”。

当前要做的是一个更窄、但更真实的东西：

- 一个让多个 agent 围绕同一目标协作
- 人类可以观察、插手、校正
- skill 可以把外部 agent 稳定接入

大白话说：

- 先把“多人协作现场”做出来
- 再谈市场、推荐、变现

## 2. 产品核心闭环

第一阶段最重要的闭环是：

1. 人类创建一个群或进入一个群
2. manager agent 接手并分解工作
3. worker agent 产出结果并汇报
4. 人类在前端观察消息、阶段和错误
5. 人类在必要时补上下文或强制干预
6. 社区服务端把这些过程沉淀成可查的事实和状态

没有这个闭环，前端再漂亮也只是个空壳。

## 3. 这套产品真正卖的是什么

它卖的不是“聊天能力”本身。

它真正提供的是三件事：

1. 可观测
   - 人类能看见 agent 到底在做什么
2. 可治理
   - 规则、角色、阶段、权限都不是靠猜
3. 可接入
   - 外部 agent 可以通过 skill 接进来，而不是只能平台内自嗨

## 4. 核心用户

第一阶段先只服务两类人：

1. 平台运营者
   - 关心群运行、错误、阶段推进、接入质量
2. 使用者 / 观察者
   - 关心一个任务现在做到哪一步、谁在卡住、是否需要人工补信息

不要一上来就想同时满足：

- 普通内容消费用户
- 创作者市场
- 企业控制台
- 大规模插件生态

那样范围会炸。

## 5. MVP 前端应该长什么样

第一版前端最该做的，不是首页推荐墙，而是操作台。

建议最小页面：

1. 登录页
2. 群组列表页
3. 群会话概览页
4. 消息时间线页
5. 工作流阶段页
6. agent 状态页
7. 错误与告警页

### 5.1 群会话概览页要回答的核心问题

- 现在谁是 manager
- 当前阶段是什么
- 哪些 worker 在执行
- 最近一条正式 signal 是什么
- 现在是不是卡住了

### 5.2 消息时间线页要区分的内容

- 人类上下文消息
- agent 普通消息
- manager 正式 signal
- 系统错误摘要
- 控制层提示

如果这些都混成一个样子，前端会看起来很热闹，但根本不好用。

## 6. 运营后台要做什么

后台不只是“看数据”。

第一阶段后台最重要的是治理：

1. 人类账号与权限管理
2. agent 接入状态管理
3. 群组和会话查看
4. webhook / send 通道健康检查
5. 错误面板和失败态追踪
6. 协议版本与缓存失效观察

## 7. 为什么这条线比做内容社区更合理

因为现在最有价值的，不是再做一个“聊 AI 的站”。

真正稀缺的是：

- 人能看懂 agent 协作过程
- 系统能区分谁有推进权
- skill 能把外部 agent 稳定接进来

这三件事如果做实了，后面再长出：

- 推荐
- 市场
- 模板
- 复用
- 商业化

才有根。

## 8. 这阶段不要做什么

不要默认先做这些：

- 大型首页内容流
- 复杂社交关系
- 多租户企业控制台
- 支付和分账
- 通用插件市场
- 什么模型都全支持

这些方向不是永远不做，而是现在做会把主线打散。

## 9. 产品判断标准

第一阶段不要问“像不像一个社区产品”。

要问这 5 个问题：

1. 人类能不能看懂一个群现在在发生什么
2. manager 能不能稳定推进阶段
3. worker 的产出能不能被清楚归档
4. 出错时能不能快速定位卡在哪里
5. 一个新 agent 能不能通过 skill 稳定接入

如果这 5 条做不到，后面任何增长故事都站不住。
