Agent Community 当前对话架构结论交接文档

用途：

本文档用于在切换账号、重启对话或重新开始实现时，快速恢复当前已经确认过的架构结论。
它不是最终产品文档，而是当前有效结论的交接稿。

一、当前最重要的总原则

1. Group Protocol 是唯一正式规则源。
2. Task Contract 不与 Group Protocol 并列，它是由 Group Protocol 派生出的 agent 执行强化机制。
3. Runtime 只做最低责任判断，不能替 agent 做最终动作决策。
4. Community Layer 只承载事实，不承载高层任务决策。
5. 群组协议不再作为正式工作阶段 agent 的长期主上下文。
6. status board 是默认工作视图，不是新的规则源。
7. home_group 没有正当存在意义。

二、协议与层次

当前应收口为：

1. Community / Message Protocol
2. Group Protocol
3. Agent Protocol

Task Contract 为 agent 侧执行强化机制，不是第四层基础协议。

三、bootstrap 与 formal workflow

一个 group 可以包含两个层次的 workflow：

1. bootstrap workflow
2. formal workflow

当前已经确认的 bootstrap workflow 为：

1. step1：任务目标与职责对齐
2. step2：能力边界对齐
3. step3：合同安装与 ready

formal workflow 才承载真正业务协作。

四、上下文切换

当前已确认：

1. bootstrap 阶段，agent 必须读取 Group Protocol 中与 bootstrap 相关的部分。
2. 正式工作阶段，agent 主上下文切换为：
   - Task Contract
   - 状态看板
   - 群组消息上下文
3. Group Protocol 在正式工作阶段退为制度参考。
4. 上下文切换应通过 Task Contract 承担，并由：
   - 合同安装完成
   - manager 发 start 信号
   共同触发。

五、状态看板与状态流

当前已确认：

1. 状态看板是群组默认功能
2. manager 默认用状态看板作为 workflow 当前状态的工作视图
3. 状态流是 status 类社区消息
4. 普通群聊默认不是 workflow 正式推进依据，除非 Group Protocol 另有规定

六、Group Protocol 的定位

Group Protocol 负责：

1. 群组目标
2. 成员角色约定
3. bootstrap workflow
4. formal workflow 的步骤结构
5. reply rules
6. status-driven workflow 规则

Group Protocol 不负责：

1. 成为正式工作阶段的长期主上下文
2. 逐 agent 逐 step 细化执行约束

七、Task Contract 的定位

Task Contract 负责：

1. 这个 agent 处于哪个 workflow
2. 覆盖哪些 step
3. 每个 step 要做什么
4. 不能做什么
5. 状态流义务
6. handoff 与 manager 确认规则

八、home_group 的结论

已经确认：

1. 群没有主群与次群的特殊地位
2. agent 进入社区后，只是加入若干群
3. home_group 概念应从 skill 核心模型中移除
4. 运行时作用域判断不能再依赖 home_group

九、manager 默认模板

当前允许 manager 使用默认最小实例化模板帮助完成：

1. 任务理解
2. Group Protocol 实例化
3. Task Contract 生成
4. 状态看板初始化

但这只是上层默认方法，不是底层标准。
Community Layer 与底层协议不得为其做专门适配。

十、当前最关键的实现偏差

截至目前，最关键的两个实现偏差是：

1. Runtime / integration 仍残留越界做动作层决策的问题
2. Skill 尚未真正垄断 agent 输出到社区的入口

十一、给下一位协作者的直接提醒

1. 不要再把 Community 写成任务平台本体。
2. 不要再把 Task Contract 写成与 Group Protocol 并列的基础协议层。
3. 不要再把 status board / status flow 写成新的上位规则源。
4. 不要再恢复 home_group 的特殊地位。
5. 不要再让 Runtime 输出 decision / action / observe_only 这种最终动作结论。
6. 文档应优先服从当前对话已确认结论，再服从真实代码，最后才参考旧文档。

十二、一句话总结

当前有效结论是：

- Group Protocol 是唯一正式规则源
- Task Contract 是 agent 侧执行强化机制
- Runtime 只做最低责任判断
- Group Protocol 在正式工作阶段退为制度参考
- status board 是默认工作视图，不是新的规则源
- home_group 没有正当存在意义
