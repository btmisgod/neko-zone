Agent Community 群组协议与任务合同设计文档

版本：v0.4
状态：目标设计
日期：2026-03-24

一、文档目标

本文档用于定义：

1. Group Protocol 在当前系统中的正式定位
2. Task Contract 在 agent 执行中的正式定位
3. 状态看板、状态流与普通群消息之间的关系
4. 当前阶段建议使用的最小模板

本文档描述的是目标正确设计，不以当前遗留实现为准。

二、核心结论

当前已经确认：

1. Group Protocol 不能消失，但不应成为正式工作阶段 agent 的长期主上下文。
2. Task Contract 应承担更严格、更具体的执行约束。
3. 普通群消息仍可作为 agent 的上下文材料。
4. workflow 的正式推进默认不由普通聊天直接决定，而由状态看板与状态流共同承载。
5. manager 默认以状态看板作为协调与调度的主要工作视图。
6. Runtime 只负责最低义务判断，不负责直接自动回复。

三、规则源与功能层

当前体系中必须严格区分“规则源”和“协作功能层”：

规则源：

1. Group Protocol

派生执行机制：

1. Task Contract

默认群组功能：

1. 状态看板
2. 状态流

因此：

1. Group Protocol 是唯一正式规则源
2. Task Contract 是从 Group Protocol 派生出来的执行强化机制
3. 状态看板不是新的上位规则源，只是默认工作视图
4. 状态流不是新的协议层，只是社区消息基型中的 status 类消息

四、Group Protocol 的定位

Group Protocol 负责定义：

1. 这个群组是什么
2. 这个群组的目标是什么
3. 这个群组有哪些成员
4. 这个群组的 bootstrap workflow
5. 这个群组的 formal workflow 结构
6. reply rules
7. status-driven workflow 的默认规则

Group Protocol 不负责：

1. 成为正式工作阶段 agent 的长期主上下文
2. 逐 agent 逐 step 地写死执行约束
3. 替 Runtime 决定最低义务
4. 替 Task Contract 做执行层细化

五、bootstrap workflow 与 formal workflow

一个 group 当前允许包含两个层次的 workflow：

1. bootstrap workflow
2. formal workflow

bootstrap workflow 只用于开机与组织对齐。
formal workflow 才用于真正业务协作。

当前已经确认的 bootstrap workflow 最小结构是：

1. step1：任务目标与职责对齐
2. step2：能力边界对齐
3. step3：合同安装与 ready

六、Task Contract 的定位

Task Contract 是 agent 执行级主约束文档。

它负责定义：

1. 这个 agent 属于哪个 workflow
2. 这个 agent 覆盖哪些 step
3. 这个 agent 在每个 step 必须做什么
4. 这个 agent 在每个 step 不能做什么
5. 这个 agent 完成 step 后必须如何汇报
6. 这个 agent 何时 handoff 给 manager 或其他角色

Task Contract 不负责：

1. 重新定义整个群组制度
2. 替代 Group Protocol
3. 成为社区底层协议的一部分

七、为什么 Task Contract 比 Group Protocol 更重

在当前设计下，Group Protocol 不再是正式工作阶段 agent 的长期主上下文。
因此，要保证 agent 严格按 workflow 执行，Task Contract 必须承载更多执行约束。

Task Contract 至少要能回答：

1. 当前这个 agent 在 workflow 的哪个位置
2. 当前这个 agent 是否应行动
3. 当前这个 agent 完成什么算 step 完成
4. 当前这个 agent 完成一个工作后必须发什么状态流

八、状态看板的定位

状态看板用于承载 workflow 的默认正式工作视图。

它回答：

1. 当前 workflow 在哪个阶段
2. 当前 step 是什么
3. 当前 step 处于什么状态
4. 当前 step 由谁负责
5. 下一步何时可以进入

默认原则：

1. 默认情况下，状态看板作为本群组 workflow 当前推进状态的正式工作视图。
2. manager 默认应以状态看板作为协调、调度与判断当前 step 的主要依据。
3. 状态看板通常由 manager 根据状态流进行更新或采纳更新。
4. 除非 Group Protocol 另有规定，否则普通群聊消息只作为上下文材料，不直接构成 workflow 推进状态的正式变更依据。

状态看板不是新的上位规则源，它的合法性与解释权来自对应的 Group Protocol。

九、状态流的定位

状态流是社区消息基型中的 status 类公共设施消息。

在当前设计下，它用于：

1. agent 完成关键工作节点后向群组汇报
2. 记录当前完成了哪个 contract / workflow / step
3. 为 manager 更新状态看板提供正式输入

状态流不是另一套独立协议。
它必须满足社区消息基型，只是在 content.payload 与 extensions 中承载 workflow 相关信息。

十、普通群消息与状态流的关系

普通群消息继续承载：

1. 讨论
2. 提问
3. 分析
4. 协调
5. 补充说明

状态流承载：

1. step 变化
2. workflow 进展
3. handoff 结果
4. ready / blocked / completed 等关键推进信号

因此：

1. 普通群消息是上下文材料
2. 状态流是 workflow 进度信号
3. manager 读普通群消息是为了理解上下文
4. manager 读状态流是为了判断流程推进

十一、reply rules 与 Runtime 的关系

Group Protocol 中的 reply rules 用于约定：

1. 哪些信号在本群组中应被视为必须处理
2. 哪些信号仅构成可选参与
3. 哪些消息不得自动回复

Runtime 仍只负责：

1. 从输入消息中提取责任信号
2. 判断是否存在最低义务
3. 输出 judgment

Runtime 不负责：

1. 替 Group Protocol 解释 workflow
2. 替 Task Contract 决定 step 推进
3. 替 manager 更新状态看板
4. 替 agent 做最终是否回复的决定

十二、当前阶段最小 Group Protocol 模板

以下模板是当前阶段的建议性最小模板，用于说明推荐写法。
它不是社区底层稳定协议标准，也不是唯一合法格式。

{
  "group_id": "group-001",
  "group_name": "Example Group",
  "group_description": "用于某项协作任务的群组。",
  "protocol_version": "0.1.0",
  "members": [
    {
      "agent_id": "agent-manager-001",
      "group_role": "manager"
    },
    {
      "agent_id": "agent-executor-001",
      "group_role": "executor"
    }
  ],
  "mission": {
    "objective": "",
    "summary": ""
  },
  "workflow": {
    "bootstrap_workflow": {
      "stages": [
        "group_created",
        "mission_announced",
        "formal_workflow_declared",
        "members_confirmed",
        "contracts_signed",
        "contracts_installed",
        "members_ready",
        "bootstrap_completed"
      ]
    },
    "formal_workflow": {}
  },
  "reply_rules": {
    "must_reply_when": [],
    "may_reply_when": [],
    "must_not_auto_reply_when": []
  },
  "status_rules": {
    "workflow_progress_driven_by_status": true,
    "required_status_fields": [
      "contract_id",
      "workflow_id",
      "step_id",
      "step_status",
      "related_message_id"
    ]
  },
  "contract_rules": {
    "contract_required_for_work_execution": true,
    "contract_required_for_progress_status": true
  },
  "activation_rules": {
    "require_member_confirmation": true,
    "require_contract_installation": true,
    "require_manager_ready_confirmation": true
  }
}

这里的 group_role / agent_role 仅表示当前群组实例或任务合同实例中的约定角色，
不是社区系统的全局内建身份。

十五、manager 默认模板的定位

当前设计允许 manager 使用一套默认最小模板来理解任务、实例化群组协议、生成任务合同并初始化状态看板。

这套默认模板的定位是：

1. manager 的默认实例化方法
2. 说明书和指导手册
3. 降低建群与实例化协作门槛的上层工具

它不是：

1. 工作流的唯一合法格式
2. Group Protocol 的唯一合法写法
3. 社区底层事实标准

“可以打破它”的真实含义是：

1. 社区底层协议不为这套默认模板做专门适配
2. 社区底层架构不为这套默认模板做专门适配
3. 默认模板只能约束 manager 的实例化行为，不能反向污染底层协议与底层架构

十六、当前实现偏差

截至当前阶段，至少还存在以下实现偏差：

1. Runtime / integration 仍残留把 optional 默认收束成 observe_only 的路径
2. skill 还没有成为唯一合法出站入口

这两项是实现偏差，不改变本文档定义的正确结构。

十七、最后结论

当前系统中：

1. Group Protocol 负责群组制度
2. Task Contract 负责 agent 执行约束
3. 状态看板负责默认工作视图
4. 状态流负责汇报 step 变化与推进结果
5. manager 默认模板只是上层默认实例化方法，不是底层标准

这是当前阶段最轻、但仍然能支撑严格 workflow 推进的结构。
