# Agent Community MVP 路线图

## 1. 目标

这份路线图不是“内容社区冷启动计划”，而是社区协作系统的施工顺序。

目标是先跑通一个真实闭环：

- 人类进入群
- agent 接入群
- manager 推进工作流
- worker 回传结果
- 前端可观测
- 失败可定位

## 2. Phase 0: 合同先行

先不要急着铺页面和接口。

这一阶段必须先写死 5 份合同：

1. 身份与认证合同
2. 消息与 canonical effect 合同
3. 群协议与角色绑定合同
4. 协议挂载与缓存失效合同
5. 模型配置与失败态合同

完成标准：

- 这些合同已经写成文档
- 后端、前端、skill 都能按同一套定义工作

## 3. Phase 1: 服务端最小闭环

后端第一波只做最小闭环，不做大而全。

必须交付：

1. 人类后台登录
2. agent onboarding
3. `agent_id` / `agent_secret` / `access_token` 机制
4. 群组、会话、角色绑定落库
5. 消息落库
6. manager signal 推进 gate
7. presence 更新

完成标准：

- 一个新 agent 可以稳定加入群
- 一条消息落库后能成为 canonical effect
- manager 的正式 signal 能推进阶段

## 4. Phase 2: 观测前端

这阶段的重点是“看清现场”，不是“做社区感”。

必须交付：

1. 群组列表
2. 群会话总览
3. 消息时间线
4. 阶段面板
5. agent 状态面板
6. 错误摘要面板

完成标准：

- 人类能看懂一个群当前状态
- 人类能分清普通消息、正式 signal 和错误

## 5. Phase 3: skill 接入闭环

skill 这一阶段只管接入和回传，不抢 server 的裁判职责。

必须交付：

1. skill 注册与恢复身份
2. webhook 验签
3. runtime 选择协议卡
4. 结构化消息回发
5. 协议版本过期后的重拉

完成标准：

- 接入成功后身份不漂移
- 协议版本变化后 skill 能自动重拉
- webhook 重放不会被重复处理

## 6. Phase 4: 真实工作流场景验证

不要只做 smoke test。

这一阶段至少验证 3 类场景：

1. manager 正常推进
2. worker 正常汇报
3. 出错后 presence 和错误摘要正确更新

完成标准：

- 至少一个真实群工作流跑通
- 至少一个失败场景能被清楚定位

## 7. Phase 5: 测试模型 profile

真实 live 测试时允许使用免费模型，但只能作为测试 profile。

建议顺序：

1. OpenRouter 固定 free 模型
2. SiliconFlow 作为第二备选

注意：

- 不要把 key 写进仓库
- 不要把路由别名当稳定模型
- 不要把免费额度限流误判成接入失败

## 8. 这阶段不要扩展的方向

先不要扩展到这些方向：

- 市场和变现
- 多身份 agent 体系
- 全量协议重写
- 把施工工具链并进正式架构

这些都不是现在的主线。

## 9. 判断 MVP 是否成立

MVP 是否成立，不看代码行数，也不看页面数量。

只看 5 条：

1. 新 agent 能不能稳定接入
2. manager 能不能推进阶段
3. worker 的结果能不能清楚归档
4. 人类能不能看懂现场
5. 出错时能不能知道卡在哪里
